Run `astro build` to validate all migrated content against the Zod schemas:

```bash
npx astro build 2>&1
```

The build will fail if any content file has frontmatter that doesn't match its collection schema. For each validation error:

1. Identify which file and field caused the error
2. Determine if it's a script bug (fix in migrate.mjs and re-run) or a one-off data issue (fix the individual file)
3. Common issues to watch for:
   - `publishDate` not coercible to Date (check date format)
   - `featureImage.src` path doesn't resolve (check image was downloaded, path is correct relative path starting with `./images/`)
   - `categories` contains non-string values (check flattening worked)
   - `videoUrl` not a valid URL (check URL format)

Also verify content quality with spot checks:
- Check 3 random blog posts: frontmatter has title, publishDate, categories, featureImage (if applicable), excerpt
- Check featured images exist on disk for those posts
- Check no WordPress URLs remain: `grep -r "joost.blog/wp-content" src/content/ | head -10`
- Check no block comments remain: `grep -r "<!-- wp:" src/content/ | head -10`
- Check no "Code language:" artifacts: `grep -r "Code language:" src/content/ | head -10`

Fix any issues found. The goal is zero build errors.

NOTE: The build may fail on missing page routes (since Phase 3 handles routing). Content validation errors (frontmatter schema) are the concern here. If build fails on routing/page issues, that is expected and acceptable -- only content validation must pass. To isolate content validation from routing, you can also run `npx astro check` which validates types and schemas without building pages.
